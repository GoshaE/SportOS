# Профессиональное документирование

## Принципы

1. **Язык**: Вся документация пишется на **русском языке**. Технические термины допускаются на английском в скобках.
2. **Аудитория**: Разработчики уровня Middle+, которые не участвовали в обсуждении. Документ должен быть самодостаточным.
3. **Структура**: Каждый документ — самостоятельная единица со своим оглавлением, контекстом и перекрёстными ссылками.

## Формат каждого документа

```markdown
# [Номер]. [Название модуля]

> Краткое описание (1-2 предложения) назначения документа.

## Содержание
[Автогенерируемое оглавление]

## 1. Контекст и цели
Зачем этот модуль существует, какую проблему решает.

## 2. Архитектурные решения
Ключевые решения с обоснованием (формат ADR):
- **Решение**: что решили
- **Контекст**: почему именно так
- **Альтернативы**: что рассматривалось
- **Последствия**: на что влияет

## 3. Детальное описание
Подробное описание с диаграммами (Mermaid), таблицами, псевдокодом.

## 4. Edge Cases и ограничения
Граничные случаи, известные лимиты, обходные пути.

## 5. Связанные документы
Перекрёстные ссылки на другие документы.
```

## Правила оформления

### Диаграммы
- Использовать **Mermaid** для всех диаграмм
- Подписи на русском, технические идентификаторы — на английском
- Каждая диаграмма должна иметь заголовок и пояснение

### Таблицы
- Использовать для структурированных данных (параметры, сравнения, списки)
- Заголовки колонок — жирные
- Не более 6 колонок для читаемости

### Код и псевдокод
- Язык блока кода указывать явно (`dart`, `json`, `yaml`)
- Псевдокод — с пометкой `pseudo`
- Реальный код — только для иллюстрации, не production-ready

### Перекрёстные ссылки
- Формат: `[см. 04-timing-engine.md, раздел 3.2](file:///path/to/04-timing-engine.md)`
- Всегда указывать конкретный раздел, не просто файл

### Глоссарий
- Каждый специфический термин должен быть в глоссарии
- При первом упоминании в документе — курсивом с пояснением
