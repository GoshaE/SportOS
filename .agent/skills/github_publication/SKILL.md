# Публикация в GitHub и CI/CD Web

Этот скилл предписывает строгий порядок действий перед отправкой (push) кода в репозиторий GitHub для приложения SportOS.

## Проблема
Серверы CI/CD (например, GitHub Actions для сборки Flutter Web) настроены так, что при сборке Flutter Web (даже по умолчанию в новых версиях) **любое** предупреждение линтера (info, warning), такое как неиспользуемый импорт (`unused_import`) или неиспользуемая переменная (`unused_local_variable`), может привести к **падению сборки с Exit Code 1**. 

Один забытый символ ломает весь пайплайн релиза!

## Обязательный Чек-лист перед `git push`

Перед тем как отправлять код на GitHub (особенно если ожидается автоматическая сборка web-версии), разработчик (или AI) ОБЯЗАН выполнить следующие шаги:

1. **Локальный анализ кода (Flutter Analyze):**
   Выполните команду в корне Flutter проекта (`/app`):
   ```bash
   flutter analyze
   ```
   
2. **Исправление всех Info и Warnings:**
   Если `flutter analyze` выдает ошибки (даже если это просто "info" или "warning" о `unnecessary_cast`, `unused_import`, отсутствующих фигурных скобках в if-выражениях, двойных подчеркиваниях `__` и т.д.), вы ОБЯЗАНЫ исправить их все до единой.
   **Цель:** терминал должен вывести `No issues found!`. Пайплайн не пропустит ничего другого.

3. **Локальная проверка сборки Web:**
   Чтобы окончательно убедиться, что код собирается под Web без сюрпризов, выполните:
   ```bash
   flutter build web --release
   ```
   *(Если в проекте используется настройка путей, убедитесь, что применяете правильный base-href, например: `flutter build web --base-href "/SportOS/" --release`)*.

Только после того как анализ чист и локальная сборка прошла успешно (`Exit code: 0`), разрешается делать `git commit` и `git push`.

## Важные замечания
- Не используйте устаревшие флаги при сборке Web (например, `--web-renderer canvaskit` объявлен obsolete во Flutter 3.22+).
- Всегда следите за чистотой импортов и переменных. Clean code = Clean build.
