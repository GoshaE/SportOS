# Работа с Git и ветками

## Коммиты (Conventional Commits)

Язык: **Русский**. Формат: `тип(скоуп): описание`

### Типы
| Тип | Описание | Пример |
|---|---|---|
| `feat` | Новая функциональность | `feat(timing): добавить финишный пост` |
| `fix` | Исправление ошибок | `fix(checkin): исправить фильтрацию по BIB` |
| `refactor` | Рефакторинг без смены логики | `refactor(ops): вынести баннер в отдельный виджет` |
| `docs` | Документация | `docs(skill): обновить стандарты кодирования` |
| `chore` | Зависимости, конфиг | `chore: добавить drift и supabase` |
| `test` | Тесты | `test(timing): покрыть TimingNotifier` |
| `style` | Форматирование | `style: dart format` |
| `build` | Сборка, CI/CD | `build: настроить GitHub Actions` |

## Именование веток
Формат: `тип/SPORT-номер_описание`

```
feat/SPORT-17_ops-dashboard
fix/SPORT-42_offline-sync
refactor/SPORT-55_clean-architecture
```

## Процесс Feature-разработки
1. Создаём ветку от `main`.
2. Вносим изменения, коммитим по стандарту.
3. PR с названием: `feat(auth): SPORT-17 Добавить авторизацию через Supabase`.
4. Описание PR:
   - Суть изменений.
   - Ссылка на задачу.
   - Скриншоты UI (если есть).
   - Список deprecated-кода (если есть).
5. После ревью — **squash commit** в `main`.
6. Удаление ветки.

## Правила коммитов
- Один коммит = одна логическая единица.
- НЕ смешивать `feat` + `fix` в одном коммите.
- Breaking changes: добавлять `!` → `feat(api)!: изменить формат TimingRecord`.
