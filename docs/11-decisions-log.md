# 11. Журнал архитектурных решений (ADR)

> Централизованный реестр всех решений, принятых в ходе проектирования SportOS. Формат: Architecture Decision Record.

---

## Хронометраж

### ADR-01: Абсолютное время в каждой отсечке
- **Решение**: каждая отсечка хранит `wall_clock + clock_offset`, а не только монотонное время
- **Контекст**: monotonic clock сбрасывается при перезапуске приложения
- **Альтернативы**: только monotonic (проще, не переживает краш), только wall_clock (подвержен ручной коррекции)
- **Статус**: ✅

### ADR-02: Мастер + Контрольный таймер
- **Решение**: одно устройство — мастер (официальное время), второе — контрольное (резервное + валидация)
- **Контекст**: два судьи = дублирование отсечек. Нужен один источник правды
- **Альтернативы**: среднее арифметическое (нестандартно), ручное утверждение (медленно)
- **Статус**: ✅

### ADR-03: Тройная репликация данных хронометража
- **Решение**: мгновенная репликация на мастер + контрольное + админ; локальные snapshot каждые 30 сек
- **Контекст**: данные хронометража невосстановимы
- **Статус**: ✅

### ADR-07: Photo-Finish — два судьи + видео
- **Решение**: кросс-валидация мастера и контрольного + 10-секундный видеоклип при каждой отсечке
- **Альтернативы**: только один судья (ненадёжно), специализированное оборудование (дорого)
- **Статус**: ✅

### ADR-08: Очередь меток — полный toolkit
- **Решение**: удаление, вставка, swap позиций, редактирование времени, смена BIB — все с обязательными комментариями в audit log
- **Контекст**: ошибки судей неизбежны, нужна возможность корректировки
- **Статус**: ✅

### ADR-09: Абстрактный адаптер оборудования (TimingHardwareAdapter)
- **Решение**: универсальный интерфейс для любого чипового оборудования (BLE / USB / Wi-Fi)
- **Контекст**: конкретное оборудование не выбрано
- **Альтернативы**: интеграция с конкретным вендором (ограничивает)
- **Статус**: ✅

## P2P Синхронизация

### ADR-04: Nearby Connections + Multipeer как транспорт
- **Решение**: `nearby_connections` пакет (автовыбор Wi-Fi Direct / BLE)
- **Альтернативы**: чистый Wi-Fi Direct (лимит 8 устройств, нет iOS), чистый BLE Mesh (100 Кбит/с)
- **Статус**: ✅

### ADR-05: CRDT с LWW + приоритеты для критичных операций
- **Решение**: LWW для обычных данных; иерархия `Finish > DNF > DNS` + уведомление судье для критичных
- **Контекст**: два судьи одновременно: один ставит Finish, другой DNF
- **Статус**: ✅

### ADR-06: Цепочка наследования авторитета
- **Решение**: Админ → Главный судья → Судья 1 → Судья 2. Failover через 60 сек
- **Альтернативы**: голосование (сложно), фиксированное устройство (хрупко)
- **Статус**: ✅

### ADR-10: Split-brain → трёхстадийные результаты
- **Решение**: Provisional → Verified (все маршалы синхронизированы) → Official (подтверждено авторитетом)
- **Контекст**: часть mesh может работать автономно
- **Статус**: ✅

### ADR-11: Hash-chain для Audit Log
- **Решение**: SHA-256 цепочка + репликация на все устройства
- **Контекст**: нужна защита от подмены записей
- **Статус**: ✅

## Мероприятие и настройка

### ADR-12: Минимальное создание + шаблоны
- **Решение**: создание мероприятия = название + дата. Остальное — через вкладки. Шаблоны для быстрого старта
- **Альтернативы**: wizard (раздражает повторно), одна длинная форма (перегружает)
- **Статус**: ✅

### ADR-13: Трассы как отдельная сущность
- **Решение**: трассы определяются на уровне мероприятия; дисциплина ссылается на трассу × кол-во кругов
- **Контекст**: одна трасса — много дисциплин. Изменение GPX обновляет все дисциплины
- **Статус**: ✅

### ADR-14: Каталог видов спорта — предустановленные + пользовательские
- **Решение**: базовый каталог в приложении (ездовой спорт, бег, лыжи...) + возможность создать свой
- **Статус**: ✅

### ADR-15: Конструктор категорий из измерений
- **Решение**: оси (пол, класс, возраст) → автогенерация комбинаций. Не все оси обязательны
- **Контекст**: гибкость: иногда М и Ж вместе, иногда раздельно
- **Статус**: ✅

### ADR-16: Стартовый порядок ≠ Протокольное разделение
- **Решение**: стартовые потоки и протокольные категории — независимые параметры
- **Контекст**: М и Ж могут стартовать вместе, но в протоколе быть раздельно (или нет)
- **Статус**: ✅

### ADR-17: Конфигурируемая форма регистрации
- **Решение**: базовые поля + настраиваемые (галочками) + пользовательские (конструктор формы)
- **Контекст**: разные мероприятия требуют разные документы
- **Статус**: ✅

### ADR-18: Лист ожидания — заморозка слота при загрузке чека
- **Решение**: загрузка чека замораживает таймаут ожидания
- **Контекст**: несправедливо наказывать атлета за задержку подтверждения админом
- **Статус**: ✅

### ADR-19: Перенос оплаты вместо возврата
- **Решение**: при отмене — оплата переносится на другое мероприятие. Финансовые возвраты — за рамками системы
- **Статус**: ✅

## Жеребьёвка и старты

### ADR-20: Автоматическая жеребьёвка с настройками
- **Решение**: автомат / ручная / комбо; по полу: вместе / раздельно; посев: случайный / ручной
- **Контекст**: BIB назначается при жеребьёвке, не при регистрации
- **Статус**: ✅

### ADR-21: BIB между днями — три варианта
- **Решение**: конфигурируется: сохранить / новая жеребьёвка / Гундерсен
- **Контекст**: на разных мероприятиях по-разному
- **Статус**: ✅

### ADR-22: Допуск к Day 2 — конфигурируемая политика
- **Решение**: три режима (strict / penalized / open) + admin override
- **Контекст**: DNF/DNS в Day 1 → разные правила допуска к Day 2
- **Статус**: ✅

### ADR-23: Перезаезд — без сдвига стартов других
- **Решение**: перезаезд в буфер между группами или новый слот. Не двигает existsing стартовые времена
- **Статус**: ✅

## Роли и безопасность

### ADR-24: QR-маршал — одноразовый + привязка к устройству + expiry
- **Решение**: QR действует один раз, привязан к deviceId, истекает по времени
- **Альтернативы**: многоразовый QR (небезопасно), постоянная ссылка (злоупотребления)
- **Статус**: ✅

### ADR-25: Динамическое переключение ролей
- **Решение**: пользователь переключается между назначенными ролями одним тапом. Админ может менять роли on-the-fly
- **Контекст**: стартёр после отправки всех → идёт на финиш
- **Статус**: ✅

### ADR-26: Роли на уровне мероприятия
- **Решение**: один судья обслуживает все дисциплины мероприятия
- **Альтернативы**: роли per-discipline (избыточно для малых мероприятий)
- **Статус**: ✅

## Инфраструктура

### ADR-27: Isar как локальная БД
- **Решение**: Isar 3.x — Flutter-нативная, zero-config, реактивные потоки
- **Альтернативы**: ObjectBox (сложнее CRDT), SQLite (нет реактивности из коробки), PowerSync (зависимость от облака)
- **Статус**: ✅

### ADR-28: Полная i18n (RU + EN на старте)
- **Решение**: архитектура поддерживает любое число языков; начинаем с RU + EN
- **Статус**: ✅

### ADR-29: Тестовое мероприятие вместо sandbox
- **Решение**: кнопка «создать тестовое мероприятие» с предзаполненными данными. Отдельный sandbox не нужен
- **Альтернативы**: отдельный sandbox-режим (дублирование всей функциональности)
- **Статус**: ✅

### ADR-30: Глобальная база атлетов + импорт
- **Решение**: единая база атлетов + импорт из Excel/CSV + экспорт. Каждое мероприятие ссылается на глобальные профили
- **Контекст**: атлет не вбивает данные каждый раз
- **Статус**: ✅

### ADR-31: 6 статусов мероприятия
- **Решение**: Черновик → Регистрация открыта → Закрыта → В процессе → Завершено → Архив
- **Статус**: ✅

### ADR-32: Уведомления — быстрые команды + свободный текст
- **Решение**: предустановленные команды для типовых ситуаций + произвольные сообщения по ролям
- **Контекст**: замена раций внутри mesh-сети
- **Статус**: ✅
