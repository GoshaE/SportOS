## 1. Контекст и цели

Изначально предполагалась чистая P2P-система (телефон-телефон). Однако полевые тесты и архитектурный аудит выявили критические уязвимости: батарея "Главного" личного телефона садится быстро, ОС убивает фоновые P2P-процессы (Bluetooth/Nearby Connections работают нестабильно с iPhone), а маршалы блокируют экраны.

**Новая архитектура — "Hybrid Sync (Offline-First с локальным Хабом / Облаком)"**.
**Требования**:
- Приложение работает 100% времени автономно, даже если связи нет вообще.
- Данные (отсечки) переживают перезагрузку устройства, убивание процесса, случайные свайпы.
- "Центр" (Хаб или Облако) нужен только для агрегации данных, а не для их эксклюзивной фиксации.

## 2. Архитектурные решения (ADR)

### ADR-04: Транспортный слой и Базовая Станция (Desktop Hub)
- **Решение**: Отказ от прямого P2P (телефон-телефон) в пользу выделенной локальной Wi-Fi сети + WebSockets. Роль "Мастера" переносится на **Ноутбук организатора (Desktop Hub)**, подключенный к мощному аппаратному Wi-Fi роутеру.
- **Обоснование**: Роутеры аппаратно держат десятки стабильных подключений и "бьют" на 200 метров. Ноутбук имеет идеальные точные часы, большой аккумулятор, экран для Главного Секретаря и порты для периферии (принтеры/ТВ). Личные смартфоны судей полностью снимают с себя нагрузку "раздачи сети" (Hotspot) и работают как "Толстые Клиенты".

### ADR-05: Offline-First (Локальная СУБД)
- **Решение**: На каждом мобильном устройстве развернута быстрая локальная база данных (Isar Database или SQLite).
- **Обоснование**: Никакой отправки "напрямую" в сеть по нажатию кнопок. Кнопка `[ФИНИШ]` сохраняет данные строго локально (за 1мс). Фоновый Синхронизатор впоследствии забирает эти данные из СУБД и сливает в Хаб/Облако (*Store & Forward*). Эта архитектура делает систему "неубиваемой".

### ADR-06: CRDT с LWW + приоритеты
- **Решение**: Last-Writer-Wins Register с иерархией приоритетов для критичных операций (паттерн сохранен из предыдущей версии P2P, но адаптирован под Облако/Хаб).

## 3. Иерархия Источников Времени и Данных
Система автоматически бесшовно "скатывается" по трехуровневой иерархии в зависимости от доступной инфраструктуры и покрытия сети на трассе.

| Тир | Источник | Статус | Транспорт | Описание |
|---|---|---|---|---|
| **Тир 1** | **Cloud Server** | Абсолютный Мастер | LTE / 3G | Высший приоритет. При наличии стабильного мобильного интернета на телефонах судей, они могут игнорировать локальный Хаб и лить данные через WebSockets напрямую в Облако (Supabase) для мгновенной Live-трансляции. |
| **Тир 2** | **Desktop Hub** | Локальный Центр | Wi-Fi (LAN) | Основной режим в тайге/горах без интернета. Телефоны через роутер находят Ноутбук, синхронизируют с ним время (PTP Handshake) и выгружают отсечки в его локальную БД. Секретарь мгновенно видит их на экране. |
| **Тир 3** | **Autonomous** | Оффлайн-Резерв | Local DB | Если Ноутбук завис/уснул/сгорел, а LTE нет. Приложение опирается на собственные аппаратные часы `CLOCK_BOOTTIME` и ранее сохраненный `Offset`. Данные пишутся только в локальную БД. |

## 4. Локальная База Данных (Offline-First)

Каждое мобильное устройство является независимой информационной единицей. Оно **не зависит** от доступности Ноутбука или Облака в момент фиксации времени участника.

**Жизненный цикл отсечки:**
1. Судья нажимает `[ФИНИШ]`.
2. Ядро берет аппаратное время (uptime iOS/Android) + высчитанный ранее `Offset` = Идеальное Истинное Время.
3. Отсечка МГНОВЕННО пишется в локальный `Isar Database`. Записи присваивается статус `pending_sync`.
4. Движок отдает команду UI: "Отсечка зафиксирована, обнови счетчик". Судья видит результат работы. Элемент интерфейса не "зависает" в ожидании ответа от сервера сети.
5. Отдельный фоновый *Sync Worker* видит новую запись `pending_sync`. Он проверяет соединение (Тир 1 или Тир 2) и в фоне стравливает ее по WebSockets. При успешном получении `ACK` от Хаба, статус записи в телефоне меняется на `synced`.

## 5. CRDT-модель данных и Криптография

### Syncable-метаданные

Каждая запись в локальной БД (на Хабе и на телефонах) содержит метаданные синхронизации и криптографическую подпись для защиты от подмены:

```dart
mixin Syncable {
  String deviceId;      // ID устройства-автора
  int version;          // Монотонный векторный счётчик (Lamport timestamp)
  DateTime updatedAt;   // Wall-clock (Истинное Время) для разрешения конфликтов
  bool deleted;         // Soft-delete (tombstone)
  String signature;     // Криптографическая подпись данных (Private Key судьи)
}
```

> **Безопасность (Blockchain & Public Key):** Как описано в *[07-roles-and-security.md](07-roles-and-security.md)*, каждая отсечка времени подписывается личным закрытым ключом (`Private Key`) устройства судьи. При синхронизации с Хабом или Облаком, сервер по открытому ключу (`Public Key`) мгновенно верифицирует, что данные действительно созданы этим судьей и не были изменены (Local DB Tampering).

### Стратегии по типу данных (CRDT)
- **TimeRecord (Отсечки)**: LWW (Last Writer Wins) по `version`, затем по `updatedAt`.
- **Entry (Заявки)**: LWW + merge по отдельным полям (например, один судья исправил имя, другой — пол).
- **Penalty (Штрафы)**: Add-only set (G-Set). Штраф отменяется только флагом `invalidated`, но не удаляется физически (Audit Log).
- **Критичные операции**: Иерархия приоритетов (Finish > DNF > DNS). "Финиш" всегда перебьет случайный "Не стартовал", если они возникли конфликтом на двух разных устройствах.

## 6. Процесс Синхронизации (Store & Forward)

Синхронизация работает в фоне. Принцип **воронки**: маршалы скидывают отсечки в Хаб, Хаб агрегирует их и переливает в Облако.

1. **Discovery (Поиск Хаба)**: Телефон подключается к Wi-Fi роутеру `SportOS_Race` и через протоколы (mDNS / Bonjour) мгновенно находит IP-адрес запущенного на Ноутбуке Desktop Hub-а.
2. **Handshake**: Телефон отправляет `HELLO {deviceId, vectorClock}`. Происходит вычисление `Offset` времени (PTP).
3. **Delta Sync**: Телефон шлет все свои накопившиеся записи со статусом `pending_sync`.
4. **Merge**: Ноутбук проверяет CRDT-правила, сливает их в свою СУБД и отдает `ACK` пакет.
5. Телефон помечает записи в собственной памяти как `synced` и уменьшает счетчик очереди.

## 7. Разрешение конфликтов

Возникают крайне редко, так как отсечки генерируются линейно (одна за другой). Имеют значение только при ручном редактировании данных (например, назначение чипа неизвестному бегуну двумя разными судьями одновременно).

**Базовое правило LWW**:
```text
if record_A.version > record_B.version → choose A
if record_A.version == record_B.version → choose by updatedAt (новейшая правка побеждает)
```

**Барьер Утверждения (Provisional -> Official)**:
Результат участника (его `RaceResult`) переходит из Предварительного (`Provisional`) в Финальный (`Official`) статус только тогда, когда Главный Секретарь на Ноутбуке нажимает кнопку "Утвердить". 
Система подсветит предупреждение, если очередь `pending_sync` (ожидающих отсечек) хоть от одного из активных маршалов в лесу не пуста — это значит, не все данные еще скачаны с трассы.

## 8. Индикация UI и Форс-мажоры (Crash Scenarios)

### Индикация Status Bar-а (Что видит судья):
Для кристального понимания ситуации в любой момент времени в интерфейсе горят два индикатора:
- **Значок Источника (Master)**: 🟢 **Облако** (LTE) / 🔵 **Хаб** (Локальный Ноутбук) / 🔴 **Оффлайн** (Связи нет вообще).
- **Счетчик Действий**: `Sync: OK` (очередь в локальной БД пуста) или `Pending: N` (ожидают отправки N пакетов-отсечек).

### Сценарий 1: Ноутбук Организатора завис / уснул (Оффлайн-режим леса)
1. Крышка закрыта. Wi-Fi сеть пропадает. Приложения маршалов издают писк и меняют индикатор на 🔴 **Оффлайн**.
2. Маршалы не паникуют и не прекращают работу! Они щелкают финишеров-спортсменов как обычно. Отсечки сохраняются в Isar DB, счетчик `Pending` растет.
3. Организатор на финише перезапускает Ноутбук. Поднимается локальный Wi-Fi.
4. Приложения маршалов мгновенно переходят в 🔵 **Хаб**, фоновый воркер стравливает все скопившиеся `Pending` данные за доли секунды. Потеря данных = 0. Потеря времени = 0 мс.

### Сценарий 2: Локальный роутер сгорел, но у судей есть 4G (Генератор выключился)
1. Роутер и Ноутбук физически отключаются. 
2. Приложения судей понимают, что Тир 2 (Wi-Fi Hub) недоступен.
3. Мобильный радио-модуль смартфона находит 4G/LTE сотовую вышку. 
4. Приложение автоматически перебазируется на 🟢 **Облако** (Тир 1) напрямую. Зрители дома продолжают видеть Live, даже если Секретарь на финише ослеп.

### Сценарий 3: Смахивание (Убийство) приложения Маршала
1. Телефон замерз, ОС жестко убила приложение `SportOS` для экономии батареи (или Судья случайно смахнул его из Recent/Multitasking).
2. При следующем запуске SportOS инициализирует локальную БД. 
3. *Все отсечки*, которые не успели улететь (были в статусе `Pending`), восстанавливаются в очереди. Кэшированный `Offset` аппаратного времени тоже живёт в памяти устройства. Судья спокойно продолжает работу с того же самого места. 
4. *Единственный сценарий безвозвратной потери отсечек* — физическое уничтожение телефона в реке или костре до его первого сеанса связи с Хабом (до очистки `Pending`).

## 9. Технические ограничения и выводы

| Параметр | Описание | Итог смены парадигмы |
|---|---|---|
| **Max клиентов Хаба** | Десятки и сотни (зависит только от мощности железа Wi-Fi роутера). | Устранение лимита в 8-10 подключений прямого P2P (Apple/Android). |
| **Радиус покрытия** | 100-200 метров вокруг финиша (на мощных роутерах до 500м). | Стабильная связь, можно использовать направленные антенны. |
| **Батарея судей** | Экономится кардинально. Смартфонам не нужно раздавать Hotspot. | Маршалам разрешено блокировать экраны "до появления первого бегуна", экономя 90% энергии на морозе. |

### Резюме:
Новая гибридная система делает SportOS по-настоящему распределенной и отказоустойчивой профессиональной хронометражной системой. Ни одно устройство не является единственной точкой отказа (SPOF) для хранения генерируемых полевых данных.
