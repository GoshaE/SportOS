## Критичные решения

### GAP-01: Серверная архитектура (дополнение к offline-first)
- **Проблема**: Онлайн-регистрация невозможна без сервера
- **Решение**: Гибридная модель «Сервер + P2P Mesh»
- **Сервер**: пользователи, мероприятия, регистрация, оплата, live-результаты, web-сайт
- **Локально**: хронометраж, оффлайн-данные гонки, P2P sync

### GAP-02: Граница сервер ↔ оффлайн
```
ДО мероприятия → Сервер = источник правды
ДЕНЬ мероприятия → "Скачать для оффлайн" → Isar
ВО ВРЕМЯ гонки → Локально + P2P Mesh
ПОСЛЕ гонки → Результаты → Сервер → Web
```

### GAP-03: Параллельные дисциплины
- **Проблема**: Несколько дисциплин могут идти одновременно
- **Решение**: Каждая дисциплина стартует в своё отведённое время. Одна финишная линия обслуживает несколько дисциплин

### GAP-04: Фильтрация BIB по дисциплине на финише
- **Решение**: Chip-фильтры над плитками BIB: «Все» | «Скиджоринг» | «Нарта-2». На плитках указана дисциплина. Номера диапазонами (1–10 = скиджоринг, 11–25 = нарта)

### GAP-05: BIB — глобальный пул
- **Проблема**: Физические номера ограничены (напр. 50 штук)
- **Решение**: BIB назначается из общего пула мероприятия. Один атлет = один номер, даже если бежит в нескольких дисциплинах. Диапазоны по дисциплинам настраиваются (1–10 скиджоринг, 20+ нарта) или последовательно

### GAP-06: Жеребьёвка → потом BIB (не наоборот)
```
1. Спортсмены в алфавитном порядке по группам
2. Жеребьёвка → определяется ПОРЯДОК СТАРТА
3. Назначение BIB из пула → номер на грудь
```
Два отдельных шага. Стартовая позиция ≠ BIB.

### GAP-07: Диапазоны BIB по дисциплинам
- Настраиваемые: последовательно (1–10, 11–25) или с пропуском (1–10, 20–35)
- Система проверяет: хватает ли номеров, нет ли пересечений
- Атлет в двух дисциплинах = один BIB (из первой дисциплины где участвует)

## Хронометраж — новые решения

### GAP-08: Принудительный ранний старт
- **Проблема**: Атлету BIB 50 надо уехать, очередь далеко
- **Решение**: Тап на атлета в стартовом списке → «Стартовать сейчас» или «Привязать к BIB X»
- **ActualStartTime** сохраняется отдельно от PlannedStartTime
- `NetTime = FinishTime - ActualStartTime`

### GAP-09: Ручная коррекция времени на финише
- Судья не успел тапнуть → тапает позже → вручную редактирует время
- **Обязательно**: запись в Audit Log с полями `originalTime`, `correctedTime`, `reason`, `bibNumber`

### GAP-10: Cutoff time (лимит прохождения)
- Настройка на уровне дисциплины: `cutoffTime` (напр. 2:00:00)
- При истечении: **автоматический DNF** + уведомление судье
- Судья может **override** (пропустить) с записью в Audit Log

### GAP-11: Tie-break — настраиваемый
| Режим | Описание |
|---|---|
| **shared** | Оба делят место (3 и 3, следующий — 5) |
| **start_order** | Кто раньше стартовал — выше |

### GAP-12: Гундерсен + tie в Day 1
- Если два атлета на одном месте → Day 2 оба стартуют **одновременно**

## Регистрация и заявки — новые решения

### GAP-13: Дозаявка после жеребьёвки
- Админ добавляет атлета вручную
- Два варианта стартовой позиции:
  - **В конец**: последний в очереди
  - **Привязка к BIB X**: стартует одновременно с указанным атлетом

### GAP-14: DNS — независимые заявки
- DNS в скиджоринге **не** влияет на нарту
- Каждая заявка (Entry) — самостоятельная сущность

### GAP-15: Групповой импорт атлетов
- Импорт из Excel/CSV для облегчения регистрации клубов

## Профиль пользователя

### GAP-16: Единый мульти-роль профиль
```
Профиль:
├── Основное: ФИО, контакт, фото, экстренный контакт
├── Собаки: кличка, чип, фото, вакцинации (привязаны навсегда)
├── Квалификации: ветеринар, судья категории X, ...
├── Роли по мероприятиям: 
│   ├── "Чемпионат Урала" → судья, атлет
│   └── "Кубок Сибири" → ветеринар
└── История: все старты, результаты, позиции
```
- Атлет создаёт аккаунт сам
- Организатор приглашает и назначает роль
- «Гостевая» запись — если регистрация вручную без аккаунта

### GAP-17: Владелец мероприятия (Owner)
- Модель как в Telegram: один Owner (может удалить мероприятие, назначить админов)
- Owner настраивает **права администраторов** (какие действия разрешены)
- Owner ≠ Admin: Owner — суперроль, Admin — настраиваемая

## Раунд 2: Полный путь Регистрация → Финиш → Публикация

### GAP-R2-01: Обнаружение мероприятия
- Атлет находит мероприятие: **ссылка** (share) + **поиск** в приложении (по городу/дате/виду спорта) + **лента** мероприятий

### GAP-R2-02: Ценообразование
- **Настраиваемо**: одна цена на мероприятие (по умолчанию) или per-дисциплина + combo-скидка
- Атлет может оплатить **одним чеком** на общую сумму

### GAP-R2-03: Модерация заявок
- **Конфигурируемо**: авто-одобрение (дефолт) или ручная модерация организатором
- Два типа оплаты:
  - **Ручной**: перевод на карту → фото чека → подтверждение организатором
  - **Автоматический** (v2+): платёжный шлюз → авто-статус «Оплачено»

### GAP-R2-04: Бронирование с таймаутом
- При подаче заявки → **бронь** на настраиваемое время (дефолт 48ч)
- По истечении: бронь снимается → уведомление следующему из waitlist (push + email)
- Загрузка фото чека **замораживает** таймаут до проверки организатором
- Организатор может **принудительно снять** бронь

### GAP-R2-05: BIB — обновлённая модель
- **Настраиваемо**: глобальный пул (один BIB на атлета, 90% случаев) **или** per-дисциплина (разный BIB в каждой)
- Организатор выбирает при настройке мероприятия

### GAP-R2-06: Подтверждение оплаты при нескольких дисциплинах
- Подтверждение **per-дисциплина** (каждая галочка отдельно)
- Кнопка **«Подтвердить все»** для удобства (если сумма совпала)
- Таймаут снимает бронь только с неоплаченной дисциплины

### GAP-R2-07: Политика возврата
- **Ступенчатая** (настраиваемая):

| Срок до старта | Возврат |
|---|---|
| > 7 дней | 100% |
| 3–7 дней | 50% |
| < 3 дней | 0% |

- Текст политики отображается при регистрации

### GAP-R2-08: Перевод на другую дисциплину
- Атлет может **перевести заявку** на другую дисциплину (до жеребьёвки)
- Оплата **сохраняется**. Разница — устно
- После жеребьёвки: отмена + новая заявка (теряет позицию)

### GAP-R2-09: Закрытие регистрации
- **Настраиваемо**: вручную или авто (дата + время при создании мероприятия)

### GAP-R2-10: Жеребьёвка — кто участвует
- **Настраиваемо**: только Confirmed (оплачены) или Registered (заявлены)
- По умолчанию: **только Confirmed**

### GAP-R2-11: Стартовый лист
- **Per-дисциплина**: каждая дисциплина — свой стартовый лист
- Публикация — по одному или все сразу (на выбор организатора)

### GAP-R2-12: Предзагрузка и актуализация
- «Скачать для оффлайн» → копия в локальную БД
- **Автоматическое обновление** при появлении интернета
- После нажатия «Режим гонки» → оффлайн = истина

### GAP-R2-13: Дозаявка на месте
- Атлет **в системе**: поиск → выбор → экспресс-регистрация (автозаполнение)
- Атлет **не в системе**: создание гостевого профиля + заполнение основных данных

## Раунд 2: День мероприятия — подготовка

### GAP-R2-14: Порядок проверок перед стартом
```
1. Мандатная комиссия → проверка документов
2. Ветконтроль → проверка собак + снаряжение
3. Получение BIB (только после ✅ ветконтроля)
4. Выбор собаки перед стартом
5. Старт
```
- Ветконтроль **до** получения BIB
- Чек-лист: ⚠️ индикатор на экране стартёра, **не блокирует** старт

### GAP-R2-15: Смена собаки в день мероприятия
- Разрешена **до старта** (по правилам ездового спорта — за 1 час)
- Организатор меняет в заявке: по чип-номеру новой собаки
- Ветеринар должен проверить новую собаку
- Стартовый лист обновляется. BIB не меняется
- **После старта / Day 2**: запрещено (настраиваемо)

### GAP-R2-16: Опоздавший атлет
- Атлет пришёл после своего планового старта
- Стартёр **отменяет DNS** → атлет стартует
- `NetTime = FinishTime - PlannedStartTime` (проигрыш — его проблема)
- На усмотрение судей: можно дать ActualStartTime (принудительный старт)

### GAP-R2-17: Конфликт расписания при нескольких дисциплинах
- Ответственность **атлета**
- Система **не проверяет** пересечения

### GAP-R2-18: Финиш — только стартовавшие BIB
- На экране финиша отображаются **только** BIB, чей старт зафиксирован
- Нет старта = нет в списке → невозможно назначить финиш не тому

## Раунд 2: Завершение и результаты

### GAP-R2-19: Все на трассе — учтены
- **Нельзя завершить дисциплину**, пока каждый стартовавший не получил статус (Finished / DNF / DNS)
- Индикатор: «⚠️ 1 атлет на трассе (BIB 42)» → alert маршалам
- Правило безопасности: если атлет «потерялся» — маршалы ищут

### GAP-R2-20: Путь результатов
```
Live (авто) → Завершить дисциплину → Provisional
→ Grace period (настр.) → Протесты закрыты
→ Главный судья утверждает → Official
```
- Пока есть **открытые протесты** → Official **заблокирован**

### GAP-R2-21: Протесты — полный workflow
```
1. Судья фиксирует: «BIB 11 протест на BIB 5» (+ причина)
2. 1 час на оформление (настраиваемо)
3. Судья вносит протест в систему (текст + причина)
4. Разбирательство → вердикт: наказание / отклонение
5. Если наказание → выбор штрафа → пересчёт протокола
6. Протест закрыт → можно утверждать Official
```

### GAP-R2-22: Выгрузка на сервер
- Оффлайн = истина для race data (таймы, штрафы, результаты)
- Нерасовые данные (описание, логотип) → LWW
- Автовыгрузка при появлении интернета

### GAP-R2-23: Экран завершения мероприятия
- Чек-лист: что ✅ сделано / что ⚠️ не сделано
- Кнопки: экспорт PDF / CSV (если не ушло авто)
- Кнопка «Завершить мероприятие»
- Для тестовых мероприятий — упрощённый чек-лист

### GAP-R2-24: Архивация
- Данные **per-мероприятие**: старые остаются в архиве
- Атлет может открыть прошлое мероприятие и посмотреть результаты
- Новое мероприятие = чистые данные

## Классификация данных для синхронизации

### Граница: кнопка «Режим гонки»

```
ДО ГОНКИ:           Сервер = источник правды
─── [Скачать для оффлайн] ───
─── [Перейти в режим гонки] ───
ВО ВРЕМЯ ГОНКИ:     Оффлайн = источник правды
─── [Завершить мероприятие] ───
ПОСЛЕ ГОНКИ:        Выгрузка оффлайн → сервер
```

### Уровни ценности данных

| Уровень | Данные | При конфликте |
|---|---|---|
| 🔴 **Священные** | TimeMark (отсечки), штрафы, DNF/DNS/DSQ, audit log | Оффлайн ВСЕГДА побеждает |
| 🟠 **Расовые** | Стартовые времена, BIB, допуск ветконтроля, выбор собаки | Оффлайн побеждает в день гонки |
| 🟡 **Организационные** | Регистрация, оплата, стартовые списки, расписание | LWW + уведомление о конфликте |
| 🟢 **Метаданные** | Описание, логотип, место, контакты | LWW без уведомлений |

## Раунд 3: Параллельные дисциплины

### GAP-R3-01: Финиш дисциплино-агностичен
- Финиш **не различает** дисциплины — судья визуально видит лыжник или нарта
- Основная задача финиша: **отметить время + назначить BIB**
- При назначении BIB: фильтр по дисциплине если нужно (но не обязателен)

### GAP-R3-02: Дисциплина важна на старте, не на финише
- Стартёр видит **общую очередь** всех дисциплин по времени
- Цветовая индикация дисциплины в очереди
- Переключение не нужно — хронологический порядок

### GAP-R3-03: Назначение судей
- Судья привязан к **роли** (старт/финиш/маршал), не к дисциплине
- Возможность привязать к дисциплине — не приоритет

### GAP-R3-04: Мастер/Контрольный таймер
- **Один** мастер + один контрольный на всё мероприятие (все дисциплины)

## Раунд 3: Многодневное мероприятие

### GAP-R3-05: Структура
- Многодневное = **одно** мероприятие, Day 2 = продолжение Day 1
- Данные и связи (суммарное время, Гундерсен) сохраняются внутри

### GAP-R3-06: DSQ Day 1 → Day 2
- DSQ в Day 1 → **не допускается** к Day 2 (по умолчанию)
- Организатор/судья может **override** (принудительно допустить)

### GAP-R3-07: Суммирование результатов
- Протокол: колонка per-day + **сумма** + средняя скорость per-day и суммарная
- Всё **конфигурируемо** (организатор включает/выключает колонки)
- Штрафы Day 1 переносятся в итоговый
- DNF Day 1 → политика (strict/penalized/open, настройка)

### GAP-R3-08: Данные между днями
- При наличии интернета → выгрузка вечером, обновление утром
- Без интернета → данные на устройствах, утром P2P sync

### GAP-R3-09: «Вне конкурса» (ВК)
- Атлет может бежать **вне конкурса** на **любом** мероприятии (не только многодневном)
- Решение за судьями/организатором
- В протоколе: отметка «ВК», не влияет на позиции в зачёте
- Пример: атлет приехал только на Day 2 → бежит ВК

## Раунд 3: Хаб и навигация

### GAP-R3-10: Главный экран атлета
- Показывать **предстоящие** мероприятия (не пустой экран)
- Если предстоящих нет → **прошедшие** с результатами
- Поиск по городу/дате/виду спорта

### GAP-R3-11: Создание мероприятия
- Кнопка «Создать мероприятие» доступна **всем пользователям**
- Создатель автоматически = Owner
- Открытая платформа — не ограничиваем
- **Тестовое мероприятие** не нужно (убрано из Feature Map)

### GAP-R3-12: Работа внутри мероприятия
- Все мероприятия видны, но внутри конкретного — работаешь там
- Можно выйти и вернуться

### GAP-R3-13: Уведомления
- Экран уведомлений (inbox) **нужен**
- Приглашения, результаты, вакцинации, протесты

### GAP-R3-14: Основная навигация
- Определить позже после сбора всего функционала

### GAP-R3-15: Видимость черновика
- Черновик виден **только** организаторам и команде
- Публичная видимость — после открытия регистрации

## Раунд 3: Эстафета (relay)

### GAP-R3-16: Формирование команд
- Организатор формирует команды **вручную**
- Количество участников — **настраиваемо** per-дисциплина
- Неполные команды **не участвуют**

### GAP-R3-17: Зона передачи
- Зона передачи есть
- Два варианта: **по касанию** или **по пересечению линии**
- Судья фиксирует момент передачи

### GAP-R3-18: Хронометраж эстафеты
```
Команда «Хаски»:
├── Этап 1: Петров  — старт 10:00:00, финиш 10:08:23 (08:23)
├── Этап 2: Иванов  — старт 10:08:23, финиш 10:17:45 (09:22)
└── Этап 3: Козлов  — старт 10:17:45, финиш 10:25:10 (07:25)
Итого: 25:10
```
- Время **каждого этапа** записывается отдельно + **общее время**

### GAP-R3-19: Штрафы и DNF/DSQ в эстафете
- Штраф участника = штраф **команды** (прибавляется к общему времени)
- DSQ/DNF **одного** участника = DSQ/DNF **всей команды**

### GAP-R3-20: Собаки в эстафете
- Каждый участник — **своя** собака (одна собака ≠ два этапа)
- Эстафета обычно **после основной гонки** → ветконтроль уже пройден

### GAP-R3-21: BIB в эстафете
- **Per-участник** (каждый бежит со своим номером)
- Номера вводятся **вручную** для эстафеты

### GAP-R3-22: Старт эстафеты
- **Масс-старт** (все команды одновременно)

## Раунд 3: Командные гонки (team scoring)

### GAP-R3-23: Командный зачёт ≠ эстафета
- Каждый бежит **индивидуально**, результаты → очки → командный зачёт
- Формирование команд: **до** гонки (при регистрации) или **после** (по договорённости) — настраиваемо

### GAP-R3-24: Системы подсчёта баллов

| # | Шаблон | Принцип | Пример |
|---|---|---|---|
| 1 | Прямая | Больше = лучше | 1→100, 2→80, 3→60... |
| 2 | Штрафная (кросс) | Меньше = лучше | 1→1, 2→2, 3→3... |
| 3 | Убывающая шкала | Нелинейная | 1→20, 2→17, 3→15, 4→13... |
| 4 | С коэффициентом | Зависит от числа команд | Баллы × k |
| 5 | Кастомная | Организатор задаёт | Любая таблица |

- Организатор выбирает шаблон и может редактировать
- Tie-break: больше первых мест → больше вторых → ...

### GAP-R3-25: Min/Max участников в команде
- Max: максимум в команде (напр. 7)
- Зачёт: лучшие N (напр. 5 из 7)
- Min: минимальный порог для зачёта (напр. 2)
- Финишировали >= min → команда в зачёте

### GAP-R3-26: Командный протокол
- **Отдельный протокол**: место команды, сумма баллов, состав
- В **личном** протоколе: доп. колонка «Очки»

## Раунд 4: Анализ конкурентов

> Исследованы: RaceSplitter, WebScorer, MyLaps, RaceGorilla, Canicompet, RaceID.
> Цель: найти функции, которые мы пропустили.

### Вердикт
SportOS покрывает 90%+ функционала конкурентов. Наши преимущества: offline/P2P (уникально), ветконтроль (уникально), управление собаками (глубже Canicompet), audit trail с hash-chain.

### GAP-R4-01: Серийные соревнования (Series/Cup)
- Серия = несколько мероприятий (этапов), объединённых в кубок/сезон
- Сквозной рейтинг: очки суммируются за все этапы
- Отсев: «лучшие 4 из 5 этапов» (настраиваемо)
- Серийный протокол: итоговый + per-этап
- Шаблоны очков (те же что для командного зачёта)
- **Приоритет: v2**

### GAP-R4-02: Transfer регистрации
- Атлет не может прийти → **передаёт** заявку другому атлету
- Оплата переносится, ветконтроль заново для нового атлета
- Организатор видит замену, запись в audit log
- **Приоритет: v2**

### GAP-R4-03: GPS Live Tracking
- Атлет берёт на трассу телефон или GPS-часы
- Зрители в приложении видят точку атлета на карте
- Организатор загружает GPX-трек маршрута
- Прогноз финиша по текущей скорости + оставшейся дистанции
- Alert безопасности: «атлет остановился N минут»
- Зритель выбирает до 10 атлетов для отслеживания
- **Приоритет: v2/v3**

### GAP-R4-04: Купоны / промокоды
- Создание промокода: скидка %, фиксированная сумма, 100% (бесплатно)
- Лимит использований (1 раз / N раз)
- Срок действия
- Отчёт: сколько использовали, кто
- **Приоритет: v2**

### GAP-R4-05: Автокатегория по возрасту
- Дата рождения → автоматическое определение категории (юниор/взрослый/ветеран)
- Организатор настраивает границы возрастов
- **Приоритет: MVP** (простая логика)

### GAP-R4-06: One-click повторная регистрация
- Атлет зарегистрировался ранее → «Зарегистрироваться как в прошлый раз»
- Данные, дисциплины, собаки подтянуты — одна кнопка
- **Приоритет: v2**

### GAP-R4-07: Alphanumeric BIB
- Поддержка буквенно-цифровых номеров: «A-05», «К-12», «SPRINT-03»
- Организатор задаёт паттерн
- **Приоритет: MVP** (влияет на модель данных — BIB = string, не int)

### GAP-R4-08: Refundable Booking
- Платная опция «возвратная бронь»: атлет доплачивает за гарантию отмены
- Настройка: % доплаты
- **Приоритет: v2**

### GAP-R4-09: Источник регистрации (Campaign tracking)
- UTM-метки при шаринге ссылки на мероприятие
- Отчёт: откуда пришли регистрации (Telegram, Instagram, сайт)
- **Приоритет: v2**

### Отклонённые
- **QR/Barcode на BIB** — ручной тап быстрее при 30-50 участниках. Пересмотреть для массового бега (500+)
- **SMS-уведомления** — платно, заменяется push + email

## Раунд 5: Глубокий аудит (забытые функции)

> Повторный анализ методик организации спортивных мероприятий.
> Найдены критические пробелы в юридической и пользовательской частях.

### GAP-R5-01: Сертификат участника / Финишер-диплом
- Автоматическая генерация PDF после утверждения Official результатов
- Данные: имя, место, время, дисциплина, мероприятие, дата, логотип
- Организатор настраивает шаблон (конструктор)
- Доступно из профиля атлета, push-уведомление
- **Приоритет: MVP**

### GAP-R5-02: Юридические документы при регистрации
- **Согласие на обработку персональных данных** — обязательная галочка
- **Отказ от ответственности** (waiver) — обязательная галочка
- **Для несовершеннолетних** — подпись родителя/опекуна
- Организатор загружает свой текст документов
- Хранение: дата + IP + факт согласия
- **Приоритет: MVP** (юридическая необходимость)

### GAP-R5-03: Согласие на фото/видео
- Отдельная галочка при регистрации
- Текст настраивается организатором
- **Приоритет: MVP**

### GAP-R5-04: Чек-ин (приход на мероприятие)
- Отметка атлета: пришёл на мероприятие (QR / тап судьи)
- Статистика: 45 из 50 пришли
- Не пришёл → информация организатору
- **Приоритет: v2**

### GAP-R5-05: Конструктор форм регистрации
- Организатор добавляет кастомные поля: размер футболки, аллергии, парковка
- Типы: текст, число, дропдаун, чекбокс, дата
- **Приоритет: v2**

### GAP-R5-06: Шаринг результатов (Social)
- Кнопка «Поделиться» → красивая карточка с результатом
- Генерация изображения: «Я занял 3 место! 🏆 00:38:12»
- Шаринг в Telegram, Instagram Stories, VK
- **Приоритет: v2**

### GAP-R5-07: Personal Best в профиле
- Лучшее время per-дисциплина
- При побитии → 🎉 «Новый личный рекорд!»
- История PB: когда, где, какое время
- **Приоритет: v2**

### GAP-R5-08: Волонтёры
- Отдельная регистрация для волонтёров на мероприятие
- Назначение на позиции (финиш, чекпоинт, регистрация)
- Координация + уведомления
- **Приоритет: v2**

### GAP-R5-09: Аналитика и отчёт для спонсоров
- Количество участников per-город, per-возраст, per-пол
- Источники регистрации
- Финансовый отчёт (собрано / расход)
- Экспорт PDF/CSV для отчёта перед спонсорами / федерацией
- **Приоритет: v2**

### GAP-R5-10: Мультиязычность
- Описание мероприятия на нескольких языках (RU + EN)
- Регистрационная форма на языке пользователя
- Протоколы на нескольких языках
- **Приоритет: v2**
